CONTEXTO DEL PROYECTO Y REQUISITOS DE INTEGRACIÓN
Necesitamos implementar un flujo de firma digital de documentos PDF integrado con la aplicación de escritorio Autofirma en un entorno web basado en Node.js (Express) en el backend y JavaScript (Frontend).El flujo de firma debe ser de tipo Monofásico (la codificación y renderizado de la firma se ejecutan en el equipo cliente a través de la aplicación nativa invocada por WebSocket local/protocolo afirma://). El script encargado de orquestar la comunicación del lado del cliente es autoscript.js y se servirá en la ruta /autoscript.js (físicamente ubicado en @public/autoscript.js en nuestro servidor).1. FLUJO DE DATOS Y OPERACIÓNEl sistema funciona bajo una lógica de sesión única sin login. El PDF activo de la sesión actual se ubica en el backend en la ruta:uploads/sessions/<sessionId>/documento.pdf (o el nombre de archivo activo de la sesión).El flujo de integración exacta que debes programar es:El usuario pulsa un botón visible de "Firmar Documento" en la interfaz del navegador.El frontend realiza una llamada HTTP GET al endpoint del servidor: /api/session/:sessionId/export-pdf.El backend lee el archivo PDF de la sesión en uploads/sessions/<sessionId>/ y lo devuelve como un binario (Blob).El frontend recibe el Blob, lo convierte a Base64 en memoria.El frontend inicializa e invoca a Autofirma pasando el documento en Base64 utilizando la función asíncrona de AutoScript.El usuario selecciona su certificado digital de firma en la interfaz de Autofirma y firma el documento.El frontend recibe el PDF firmado resultante en Base64.El frontend ofrece de manera inmediata la opción de descargar el archivo firmado localmente o, en su defecto, enviarlo de vuelta al servidor mediante una petición POST al endpoint /api/session/:sessionId/import-signed-pdf para re-importarlo y sobrescribir el archivo activo con su versión firmada.2. ESPECIFICACIONES TÉCNICAS REQUERIDASA. REQUISITOS DEL BACKEND (Node.js / Express)Debes implementar o actualizar los siguientes endpoints y middleware:GET /api/session/:sessionId/export-pdf:Validar que existe el directorio uploads/sessions/<sessionId>/ y que contiene un archivo PDF válido.Enviar el archivo como un flujo binario con el tipo MIME application/pdf.POST /api/session/:sessionId/import-signed-pdf:Recibir un cuerpo JSON con { signedPdfB64: "..." }.Decodificar la cadena Base64 recibida a binario.Guardar/sobrescribir de forma segura el archivo en la ruta de la sesión (uploads/sessions/<sessionId>/documento_firmado.pdf o reemplazar el original).Retornar un estado HTTP 200 OK con un mensaje JSON de éxito.Cabecera Content-Security-Policy (CSP):Configurar o ajustar el middleware de seguridad (como helmet o cabeceras manuales) para que el navegador permita la llamada al protocolo externo de Autofirma. Debe incluir el esquema afirma://* en la directiva default-src o connect-src.Ejemplo: Content-Security-Policy: default-src 'self' afirma://*; (Asegúrate de no romper otros scripts inline o estilos necesarios).B. REQUISITOS DEL FRONTEND (JavaScript / HTML)Implementa un módulo o script de cliente que cumpla estrictamente con las siguientes reglas del manual de Autofirma:Carga e Inicialización:Importar /autoscript.js de forma asíncrona o en el <head>.Llamar a AutoScript.cargarAppAfirma() una única vez tras la carga de la página para preparar el WebSocket de comunicación local.Garantizar la interacción del usuario (Crucial):La llamada de firma AutoScript.sign(...) nunca debe ejecutarse de forma automática. Debe activarse estrictamente dentro del manejador de eventos click del botón de firma para evitar que navegadores como Google Chrome u otros bloqueen la invocación al protocolo externo por motivos de seguridad.No concurrencia:Bloquear o deshabilitar el botón de firma (y mostrar un spinner/diálogo de carga personalizada) inmediatamente tras el clic del usuario. Esto evitará que se lancen múltiples llamadas paralelas a Autofirma antes de recibir una respuesta.Conversión y Parámetros del Método sign:Convertir el Blob recibido del backend a una cadena Base64 limpia.Llamar a la función:AutoScript.sign(
    pdfBase64,           // Datos a firmar en Base64
    "SHA256",            // Algoritmo de firma (bajo normativas ENS/SHA-2)
    "PAdES",             // Formato de firma para PDFs
    null,                // Parámetros adicionales (null por defecto)
    successCallback,     // Callback de éxito
    errorCallback        // Callback de error
);
Manejo de Respuestas (Callbacks):successCallback(signatureB64, certificateB64):Ocultar el spinner de carga.Realizar la petición POST a /api/session/:sessionId/import-signed-pdf para guardar el PDF firmado en el servidor.(Opcional) Proporcionar un enlace o botón para descargar el archivo firmado directamente en el navegador del cliente utilizando un objeto Blob y window.URL.createObjectURL(blob).errorCallback(errorType, errorMessage):Ocultar el spinner de carga.Mostrar un mensaje de error elegante y amigable en pantalla (evita usar alert(), usa componentes de alerta UI del sistema).3. TAREAS QUE DEBES REALIZAREstructura de Carpetas: Asegura que el archivo @public/autoscript.js esté correctamente referenciado.Desarrollo Backend: Crea/modifica el controlador de Express con los endpoints especificados y el manejo de errores correspondientes (archivos inexistentes, decodificación corrupta de Base64, etc.).Desarrollo Frontend: Escribe el archivo JavaScript/HTML de la vista que contendrá el botón, el spinner de carga, la llamada al endpoint para obtener el PDF, la lógica de conversión a Base64, la invocación asíncrona a AutoScript.sign, el callback para enviar la firma de vuelta al backend y los descargables.Manejo de Errores e Indicadores de Carga: Implementa un flujo visual claro para que el usuario sepa que la aplicación de Autofirma se está abriendo localmente y que debe interactuar con ella.
